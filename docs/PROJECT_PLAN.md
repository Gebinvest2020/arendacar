# PROJECT_PLAN — Сайт аренды автомобилей

## Цель проекта

Конечная цель — **вариант B: рабочий MVP** с базой данных, заявками и простой административной панелью.

Разработку начинаем с **визуальной публичной версии на тестовых данных**, чтобы сначала показать её заказчику и согласовать внешний вид, и только потом подключать базу, заявки и админку.

Стек: Next.js (App Router), TypeScript, Tailwind CSS, PostgreSQL, Prisma, Zod, React Hook Form, Vercel, хранилище для фото, Telegram-уведомления менеджеру.

---

## Фазы разработки

Каждая фаза — это законченный, проверяемый шаг. Переход к следующей фазе только после проверки и (при разрешении заказчика) commit.

### Фаза 1 — Публичная визуальная основа

**Что делаем:**
- общий дизайн (палитра, шрифты, базовые UI-компоненты);
- шапка (header);
- мобильное меню;
- главная страница;
- форма выбора города и дат (получение / возврат);
- блок категорий;
- блок популярных автомобилей;
- блок преимуществ;
- схема аренды (как это работает);
- условия;
- контакты;
- футер.

**Данные:** тестовые (моки), без базы.

**Проверка:** `npm run lint`, `npm run build`, визуальный просмотр на десктопе и мобильном.

**Commit после фазы:** да.

---

### Фаза 2 — Каталог и страница автомобиля (на тестовых данных)

**Что делаем:**
- каталог автомобилей;
- фильтры;
- сортировка;
- карточки автомобилей;
- отдельная страница автомобиля;
- галерея;
- характеристики;
- тарифы (цены по срокам);
- залог;
- предварительный расчёт стоимости (пока на клиенте, на тестовых данных).

**Данные:** тестовые (моки), без базы.

**Проверка:** переходы по каталогу и карточкам, работа фильтров/сортировки, адаптив, `lint` + `build`.

**Commit после фазы:** да.

---

### Фаза 3 — PostgreSQL и Prisma

**Что делаем — модели данных:**
- города;
- категории;
- автомобили;
- фотографии;
- тарифы;
- заявки.

Подключение PostgreSQL, схема Prisma, миграции, тестовые данные (сиды). Публичные страницы начинают читать данные из базы вместо моков.

**Проверка:** миграции применяются, данные видны (Prisma Studio), страницы отображают данные из базы.

**Commit после фазы:** да (файл `.env` в git НЕ попадает).

---

### Фаза 4 — Рабочая форма заявки

**Что делаем:**
- серверная валидация данных (Zod);
- серверный расчёт цены (нельзя доверять цене из браузера);
- сохранение заявки в базу;
- экран успешной отправки («Спасибо, с вами свяжутся»).

**Проверка:** заявка сохраняется, серверный расчёт совпадает с показанным пользователю, подделанную цену сервер не принимает.

**Commit после фазы:** да.

---

### Фаза 5 — Доступность автомобилей

**Что делаем:**
- проверка пересечения по датам с уже подтверждёнными бронированиями;
- новая заявка сама по себе автомобиль **не блокирует**;
- автомобиль считается занятым только после перевода заявки в статус **CONFIRMED**.

**Проверка:** авто с подтверждённой бронью не показывается/не бронируется на пересекающиеся даты; новые (неподтверждённые) заявки доступность не меняют.

**Commit после фазы:** да.

---

### Фаза 6 — Telegram-уведомления

**Что делаем:**
- уведомление менеджеру о новой заявке;
- токен и chatId **только через `.env`** (никогда не в коде и не в git);
- ошибка при отправке в Telegram **не должна отменять** сохранение заявки (заявка сохраняется в любом случае).

**Проверка:** тестовая заявка → сообщение в Telegram; при недоступном Telegram заявка всё равно в базе.

**Commit после фазы:** да.

---

### Фаза 7 — Простая административная панель

**Что делаем:**
- вход администратора;
- управление автомобилями;
- управление ценами;
- управление фотографиями;
- список заявок;
- изменение статуса заявки.

Пока — один администратор, без сложных ролей.

**Проверка:** без входа админка недоступна; изменения из админки отражаются на публичном сайте; статусы заявок меняются.

**Commit после фазы:** да (можно разбить на под-коммиты: вход, CRUD, заявки).

---

### Фаза 8 — SEO, мобильная проверка и деплой на Vercel

**Что делаем:**
- метатеги, `sitemap.xml`, `robots.txt`, OpenGraph;
- финальная проверка мобильной версии (Lighthouse);
- деплой на Vercel.

**Проверка:** корректные превью ссылок, приемлемый Lighthouse (mobile), сайт открывается на Vercel.

**Commit после фазы:** да.

---

## Что специально НЕ делаем на первом этапе

- онлайн-оплату;
- автоматическое списание денег;
- сложную CRM;
- несколько ролей администратора;
- мультиязычность;
- несколько валют;
- автоматическое определение города по IP;
- сложные акции и промокоды;
- загрузку фотографий (до отдельной согласованной фазы).

## Данные, которые нужно получить от заказчика

- название компании;
- логотип;
- фирменные цвета;
- города (список);
- список автомобилей;
- фотографии;
- характеристики автомобилей;
- цены по срокам аренды;
- суммы залогов;
- условия аренды (текст);
- телефон;
- мессенджеры;
- адрес;
- куда отправлять заявки (личный Telegram или групповой чат);
- нужна ли украинская и английская версии;
- нужна ли в будущем онлайн-оплата.

## Принципы работы

- изменения — только в папке `D:\5.0\arendacar`;
- перед изменениями проверять `git status`;
- не использовать `git add .`, добавлять файлы явными путями;
- секреты и `.env` в git не попадают;
- после изменений — `npm run lint` и `npm run build`;
- commit и push — только с разрешения заказчика.
